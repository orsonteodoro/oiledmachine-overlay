The Linux kernel had several zero day vulnerabilities:

CVE-2026-43284, CVE-2026-43500 - Dirty Frag:  privilege escalation, attack chain primitive
CVE-2026-31431 - Copy Fail:  priviliege escalation, container escape, attack chain primitive

CVE-2026-43284 - CVSS:3.1/AV:L/AC:H/PR:L/UI:N/S:C/C:H/I:H/A:H (denial of service, data tampering, information disclosure)
CVE-2026-31431 - CVSS:3.1/AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H (denial of service, data tampering, information disclosure)

The AV:L (locally ran) is deceptive, it can be chained together with another
exploited vulnerability with remote code execution for full system compromise.
The chances increase if you haven't updated your kernel for months or years or
through unpatched red zone apps.

Both are mitigated.  The official patch fix for Dirty Frag is currently
partially done, so disabling the affected module is the mitigation fallback.

Always update the initramfs with updated kernel modules to avoid getting a
black bootup screen.

Remember to keep backup kernels.  Do not delete old kernels until you verify
that the kernel works.

All solutions assume that after the the kernel as been installed, a new updated 
initramfs generated for the kernel flavor, boot config updated; a reset is done.
After reset the disabled .config solution is done; but for the external module
techniques they should either prevent the loading of the affected code or
require explicitly unloading every reboot until the official fix is applied.

======================
Remediating Dirty Frag
======================

There are 3 solutions.  Solution 2 has 2 parts.

Solution 1:

     The easiest way is to use ot-sources ebuild on config assist mode.  The
     ot-sources currently uses solution 2 automatically.  If you use your own
     custom kernel or the distro you need to follow one of the the solutions
     below.  

Solution 2:

     Verify the following kernel versions for partial/full mitigation.  The
     exact numbers are currently partial but later will be full mitigation.

     Part A:  Get the patched version(s)

          >= 7.0.5
          >= 6.18.28
          >= 6.12.87
          >= 6.6.138
          >= 6.1.172
          >= 5.15.206
          >= 5.10.255

     Part B:  .config edit or unload module, your choice

          Disable or comment out in .config for the latest point releases if you
          have the above point releases or newer.

               CONFIG_AF_RXRPC

                    or 

          Unload rxrpc external kernel module.

               modprobe -r rxrpc

                    or 

          Blacklist rxrpc kernel external kernel module by editing
          /etc/modprobe.d/blacklist.conf with the following contents:

               blacklist rxrpc

Solution 3:  Earlier kernel point releases are being used but choose either the
.config edit, unload external module, or blacklist external module.

     You need all three disabled in the .config:

          CONFIG_AF_RXRPC
          CONFIG_INET_ESP
          CONFIG_INET6_ESP

               or

     Unload all affected external modules:

          modprobe -r esp4 
          modprobe -r esp6
          modprobe -r rxrpc

               or

     Blacklist all affected external modules by editing
     /etc/modprobe.d/blacklist.conf with the following contents:

          blacklist esp4
          blacklist esp6
          blacklist rxrpc

The .config disable is preferred to eliminate the vulnerable time window before
use and easier clean up after upstream fixes it.

Solution 1/2 are the most preferred because it also triages several the low level
vulnerability classes.  Solution 3 triages the in the wild but viable for those
with custom modules or slow kernel module vendors.

To verify that the external kernel module is unloaded, inspect
`cat /proc/modules`.


=====================
Remediating Copy Fail
=====================

For the Copy Fail, you need at least the following for full mitigation:

>= 7.0.5 (stable):  Patched
>= 6.18.28 (LTS):  Patched
>= 6.12.87 (LTS):  Patched
>= 6.6.138 (LTS):  Patched
>= 6.1.172 (LTS):  Patched
>= 5.15.206 (LTS):  Patched
>= 5.10.255 (LTS):  Patched

Still vulnerable hunks were observed:

<= 6.18.21 (LTS):  Not patched
<= 6.12.84 (LTS):  Not patched
<= 6.6.136 (LTS):  Not patched
<= 6.1.169 (LTS):  Not patched
<= 5.15.203 (LTS):  Not patched
<= 5.10.253 (LTS):  Not patched

Both Copy Fail and Dirty frag should be mitigated ASAP before the KEV
remediation deadline on the 15 of this month, but it is recommend earlier if
meeting billing commitments or transfering very sensitive data before this time.

When the issue is patched by upstream, you should remove the blacklists.  You
should take notes of the changes you made to undo them.
